Read this first: the truth about “all properties”

HubSpot properties are not one fixed universal list. Each portal can have different default properties, subscription-based properties, integration-created properties, and custom properties. So this file gives you the important admin-facing properties for each object and the exact admin habit you need: export the full property list from the live portal before doing implementation work.

  1. Go to Settings → Properties.
  2. Select the object, such as Contact, Company, Deal, Ticket, or Line Item.
  3. Review property groups, field types, internal names, usage, and whether the property is default or custom.
  4. Export properties before migration, automation design, cleanup, reporting, or integration work.

The simple mental model

Object = type of record. A record stores data in properties. Associations connect records to show relationships. Activities show what happened. Pipelines show process movement. Reports turn all of this into visibility.

Example: A Contact works at a Company. The Company has a Deal. The Deal has Line Items and a Quote. The customer later opens a Ticket. Calls, emails, meetings, notes, and tasks create the timeline across those records.

Top

A contact is an individual person your business knows, sells to, markets to, supports, or communicates with.

How this object is used

A visitor downloads a pricing guide. HubSpot creates a Contact with name, email, job title, source, and form submission activity. Sales can follow up and marketing can enroll them in a nurture workflow.

Mini case study

Problem: A sales team keeps importing the same people from spreadsheets, causing duplicate outreach and messy reporting.

CRM admin solution: Use email as the primary identity anchor, set import rules, review duplicate contacts weekly, standardize required fields for lead creation, and train reps to update lifecycle stage and lead status correctly.

Best practice: Do not create a contact unless there is a real person behind the record. For B2B, associate the contact to a company as early as possible.

Mini case study

Problem: A business tries to force shipments, vehicles, or projects into Deals or Tickets, causing messy data and weak reporting.

CRM admin solution: Design the schema before creating it: define object purpose, primary key, associations, pipeline needs, permissions, required fields, and reporting use cases.

Best practice: Do not create a custom object if a standard object or property can solve the problem. Custom objects are powerful but add complexity.

CRM admin best-practice checklist

  • Do not activate objects casually. More objects mean more training, permissions, reporting, and cleanup.
  • Define ownership rules. Every serious object needs a clear owner field or assignment process.
  • Separate process from data. Properties store data; pipelines show movement; workflows automate action.
  • Use associations intentionally. Bad associations create bad reporting even if properties are correct.
  • Use naming conventions. Deal, campaign, product, and custom object names should follow a pattern.
  • Protect internal names. Internal names matter for integrations and APIs. Think before creating properties.
  • Audit before automating. Automation on messy data creates faster mess.
